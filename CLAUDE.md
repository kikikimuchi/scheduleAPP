# scheduleAPP - 作業ルール

## 食材・栄養データの登録について（最重要）

- **憶測で数値を決めない。** カロリー・糖質・脂質・タンパク質・内容量（缶/瓶/mlなど）を勝手に推測して確定値のように登録しない。
- 値がわからないときは**登録前に必ず調べる**（商品パッケージの栄養成分表示、メーカー公式、店舗公式サイトなど一次情報を優先）。
- 調べても確定値が存在しないもの（生の果物・量り売りなど）だけ、**「推定値」と明示**したうえで、根拠（換算元の100gあたりの値や想定グラム数）を添えて登録する。
- 提供形態（缶・瓶・コップ・サイズ）も推測しない。実際の提供形態を確認してから数値を出す。
- ユーザーに指摘されてから調べるのではなく、**出す前に確認する**。

理由: 栄養の数字はユーザーのダイエット記録そのもの。不正確な値は記録の信頼性を損なう。

## アプリ構成メモ

- Firestore プロジェクト: `keiriauto-6f8f1`（OPEN ルール、REST で直接書き込み可）
- 食材マスターのコレクション名は **`schedule_foodMenus`**（`foodMenus` ではない）。ドキュメントIDは食材の `id`（数値）、フィールド: `id, category(main|snack|drink), name, yomi, kcal, carbs(糖質), fat, protein`。
- 体重は `schedule_weights`、ドキュメントID = 日付（例 `2026-06-25`）、フィールド: `date, weight, bodyFat` 等。
- 検索は name＋yomi をひらがな正規化して部分一致。カタカナ商品はひらがな読みを `yomi` に入れる。
- GitHub Pages は `main` ブランチからデプロイ。変更はフィーチャーブランチと `main` の両方に push する。
